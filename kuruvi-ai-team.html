I Built a Self-Managing AI Team That Runs on $2/Day

How switching to the "expensive" AI model actually saved me money

Last month, my AI bill hit $847. This month? $67. Same workload. Better results.

The counterintuitive trick? I stopped trying to save money on AI calls and let the AI manage itself.

The Problem: AI Agents Are Expensive... Until They're Not

I built Nanoclaw—a WhatsApp-powered AI assistant—because I wanted to delegate everything. Email, code reviews, research, you name it. Text it to a bot, get it done.

It worked. Too well. Within weeks, I had multiple agents running constantly:

  • Helix (main orchestrator) - My personal assistant
  • Turing - Writes production code
  • Jony - Handles UI/UX design
  • Nitty - Runs comprehensive QA tests
  • Ada - System architecture decisions
  • Steve - Product strategy

The bill was brutal. Every agent made its own API calls. No coordination. Lots of retries. Claude Sonnet at $3 per million tokens adds up fast when you're running six agents 24/7.

The Breaking Point: I watched Turing retry a bug fix 7 times because Sonnet kept hallucinating function signatures. Each retry cost me $0.42. The fix should've cost $0.06.

The Insight: More Expensive AI = Fewer Retries = Lower Costs

Here's the math that nobody talks about:

A $0.15 API call that works on the first try is cheaper than three $0.06 calls that fail.

So I did something crazy. I switched everything to Claude Opus 4.6—the most expensive model Anthropic offers.

Building Kuruvi: The Orchestration Layer

But switching models alone wasn't enough. I needed agents to coordinate. Enter Kuruvi (Tamil for "bird"—because these agents fly).

The Architecture

Kuruvi is a task queue system with a twist:

  1. Convex Backend - Real-time task queue, no polling
  2. Docker Containers - Each agent runs in isolation
  3. Discord Notifications - Agents report completions to a webhook
  4. Helix (Orchestrator) - Monitors Discord, kicks off next tasks automatically

When I text Helix "Fix the login bug on LaunchLog," here's what happens:

1. Helix creates a Kuruvi task → assigns to Turing
2. Turing spins up in Docker, reads the codebase
3. Writes the fix, commits to GitHub
4. Posts completion to Discord #agent-workspace
5. Helix sees the notification → creates QA task for Nitty
6. Nitty tests the fix, screenshots results
7. Posts to Discord → Helix verifies and deploys

All of this happens while I'm asleep. Zero manual intervention.

The Secret Sauce: Fresh Sessions Per Task

Each agent gets a fresh Claude session for every task. No context pollution. No "but earlier you told me..." confusion. Just the task description and the necessary files.

This is why Opus works so well—it can handle complex context without retries. Sonnet would need 2-3 attempts to get Turing's coding tasks right. Opus nails it first try, 98% of the time.

The Results: Quality Over Quantity

January (Sonnet, no orchestration):
• 127 tasks completed
• 381 total API calls (avg 3 retries per task)
• Cost: $847
• Success rate: 64%

February (Opus, with Kuruvi):
• 119 tasks completed
• 122 total API calls (1.03 calls per task)
• Cost: $67
• Success rate: 98%

Same developer (me). Same workload. 92% cost reduction.

Why This Matters for Everyone

You don't need a team of AI agents to apply this lesson. The principle scales:

  • For solopreneurs: Stop using ChatGPT-3.5 for everything. Use GPT-4 for critical decisions. Fewer mistakes = less rework.
  • For dev teams: Let AI agents handle the boring stuff (QA, docs, refactoring). Humans focus on product decisions.
  • For enterprises: Task queues + model selection based on complexity = massive savings at scale.

The Counterintuitive Lessons

1. Pay for quality upfront. Retries are hidden costs. A task that takes 3 attempts costs 3x as much, even if the model is cheaper.

2. Let AI manage AI. Helix orchestrates the other agents. It decides who gets which task, monitors completions, and kicks off follow-ups. I just review the final output.

3. Fresh context beats long conversations. Each Docker container is ephemeral. No "memory drift." No hallucinations from unrelated prior tasks.

What's Next: The $1/Day Target

Current bottleneck: Helix runs 24/7 monitoring Discord. That's $40/month right there.

The fix? Event-driven architecture. Helix only wakes up when Discord pings it. With Convex webhooks, I can get that cost down to under $30/month total.

Target: Six autonomous AI agents running my entire workflow for less than the price of a Netflix subscription.

Open Questions

I'm still experimenting with:

  • Agent specialization: Should Turing handle all code, or split into backend/frontend agents?
  • Memory systems: Right now agents are stateless. Should they have long-term memory for recurring tasks?
  • Human-in-the-loop: When should agents ask for approval vs. just ship it?

The Bottom Line

Most people optimize for the wrong thing. They chase cheap API calls and end up with expensive failure loops.

I optimized for success rate and let the costs follow. Turned out, success rate is the only metric that matters.

Kuruvi isn't open source (yet), but the principles are:

  1. Use the best model for the job, even if it's expensive
  2. Let AI orchestrate AI—humans are bottlenecks
  3. Fresh context > long conversations
  4. Measure retries, not just raw API costs

If you're burning cash on AI and not getting results, you're optimizing the wrong variable.
